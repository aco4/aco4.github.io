Ryan Parker, Anthony Gu, Jason Yu

Introduction¶

For our final CMSC320 Introduction to Data Science project, we are participating in the Spotify Million Playlist Dataset Challenge.

The Million Playlist Dataset (MPD) contains 1,000,000 playlists created by users on the Spotify platform. These playlists were created during the period of January 2010 through October 2017. Each playlist in the MPD contains a playlist title, the track list (including track metadata) editing information (last edit time, number of playlist edits) and other miscellaneous information about the playlist.

The Spotify MPD Challenge is a continuation of the RecSys Challenge 2018, which ran from January to July 2018. The evaluation task is automatic playlist continuation: given a seed playlist title and/or initial set of tracks in a playlist, to predict the subsequent tracks in that playlist.

This is an open-ended challenge intended to encourage research in music recommendations. The Digital Music Alliance, in their 2018 Annual Music Report, state that 54% of consumers say that playlists are replacing albums in their listening habits. And according to Spotify,

...users dont love just listening to playlists, they also love creating them. To date, over 4 billion playlists have been created and shared by users. People create playlists for all sorts of reasons: some playlists group together music categorically (e.g., by genre, artist, year, or city), by mood, theme, or occasion (e.g., romantic, sad, holiday), or for a particular purpose (e.g., focus, workout). Some playlists are even made to land a dream job, or to send a message to someone special.

By learning from the playlists that people create, we can learn all sorts of things about the deep relationship between people and music. Why do certain songs go together? What is the difference between Beach Vibes and Forest Vibes? And what words do people use to describe which playlists?

By learning more about nature of playlists, we may also be able to suggest other tracks that a listener would enjoy in the context of a given playlist. This can make playlist creation easier, and ultimately help people find more of the music they love.

# Loss function, closer to 0 is better - May want to change later
def loss(playlist, recommendations):
    temp = set(playlist.copy())
    for song in recommendations:
        if song in temp:
            temp.remove(song)
    return len(temp)/len(playlist)

def totalLoss(playlists, func):
    total = 0
    for playlist in playlists:
        total += loss(playlist[1], func(playlist[0]))

As you can see, our current method isn't particularly good. It only guesses the most popular songs, meaning that it ignores the rest of the dataset. We aren't using our resources effectively now. In fact, after 100 recommendations, our technique on average gets only 4% of the remaining songs in the playlist.

So now we must retry, and think of a new approach to recommending songs. One idea is that we can also see which songs are commonly put in the same playlist as each other. This would be done by iterating through the songs twice and using a 2D dictionary to increment whenever two songs are in the same playlist as each other. This would give us a matrix that shows common songs, which we could then use to find the songs that are most likely to be in a new playlist.

However we may run into some potential Issues:

  • May disregard less popular songs (TBD how to fix)

  • Could be quite slow

def genPairedSongs():
    songs2 = {}
    iters = 100 # Iterate through iters * 1000 playlists, around 15 seconds per iter
    #iters = 1 # Uncomment for small test runs
    x = 0
    # Iterate through files in the folder
    for filename in os.listdir(path_to_data):
        if x >= iters:
            break
        x += 1
        file_path = os.path.join(path_to_data, filename)

        f = open(file_path)
        data = json.load(f)
        f.close()
        playlists = data.get('playlists', {})

        for playlist in playlists:
            for song in playlist.get("tracks"):
                val = song.get("track_name")
                if val not in songs2:
                    songs2[val] = {}
                for song2 in playlist.get("tracks"):
                    val2 = song2.get("track_name")
                    if val2 not in songs2[val]:
                        songs2[val][val2] = 1
                    else:
                        songs2[val][val2] += 1


    # Removing very small tracks
    newsongs2 = {}
    for song in songs2:
        newsongs2[song] = {}
        for song2 in songs2[song]:
            if songs2[song][song2] > 10:
                newsongs2[song][song2] = songs2[song][song2]
        if len(songs2[song]) == 0:
            newsongs2.pop(song)

    # Saving the above in a text document to save time
    with open("SharedMatrix.txt", 'w') as f:
        json.dump(newsongs2, f)

def pairedRecommendations(playlist, amt = 100):
    playlist = set(playlist)
    scores = {}
    songs = []
    for song in playlist:
        if song in pairedSongs:
            for (name, score) in pairedSongs[song].items():
                if name not in scores:
                    scores[name] = score
                else:
                    scores[name] += score
    scores = sorted(scores.items(), key=lambda item: item[1], reverse=True)
    x = 0
    while len(songs) < amt and x < len(scores):
        if scores[x][0] not in playlist:
            songs.append(scores[x][0])
        x += 1
    return songs

Unfortunately, the algorithm seemed to group nearly every playlist under the same cluster. In fact, clusters 0 and 1 each only contain one playlist. Our hypothesis for why this was happening was because many playlists don't share a significant amount of songs, placing them closer together.

In addition, it was tricky to run predictions with this model since this is a supervised learning algorithm. We were unsure how to "correctly" label each playlist.

As a result, we had to abandon the kNN approach and try something different.

Apriori Algorithm and Association Rule Learning/Mining¶

This is not an algorithm that we learned about in class, but it is a type of unsupervised learning, which we did learn about. Association Rule Learning is commonly used with Market Basket Analysis, which is where products are recommended to customers based on what they already have. For example, Amazon will recommend you additional items you may be interested in based on items in your cart.

We realized that this was very similar to Spotify playlists. Using the Apriori algorithm which implements Association Rule learning, we would be able to solve our problem.

Handle Duplicates¶

We then realized that Spotify sometimes has multiple listings for the same song, for example sometimes in different albums. So, the number of unique tracks doesn't actually reflect the number of unique songs. To address, this, we will need a more complicated system where we keep track of songs not only by their track_uri but also by their artist.

In [22]:
class Song:
    def __init__(self, track_uri, track_name, artist_uri, artist_name):
        self.track_uri = track_uri
        self.track_name = track_name
        self.artist_uri = artist_uri
        self.artist_name = artist_name

    def __eq__(self, other): 
        return (self.track_name == other.track_name) and (self.artist_uri == other.artist_uri)
    
    def __hash__(self):
        return hash((self.track_name, self.artist_uri))
    
    def __repr__(self):
        return f'{self.track_name}{self.artist_name}'

Interpretation¶

Through our analysis and modeling, we were able to create a satisfactory prototype for a machine learning model to predict song recommendations. With an average score of 0.19, we felt that our model, while weak, was far from failure. The Apriori Algorithm was definitely the right approach to the task, with much greater success than our initial kNN approach. However, our journey was not without several significant challenges:

Collaboration¶

We relied on local Jupyter Notebook installations to run our code, because other collaborative services such as Google CoLab didn't allow uploading 31GB of data to the workspace environment. Because of this, sharing code among collaborators was difficult.

Dataset size¶

The enormous size of the dataset (31GB) posed a significant challenge. Operating within memory limits throttled certain procedures, and we had to be creative in the way we handled data. Unfortunately, we were unable to train on the entire 1 million playlists and could only use a small subset (1000). We considered using batch algorithms to decrease max memory loads, but this was something we were unable to accomplish due to the next challenge.

Time¶

Given our time constraints on this project, we were unable to achieve everything that we wanted to. For example, using batch algorithms and testing various values for our min_support hyperparameter would have been ideal, but this would have taken too long.

Insights¶

Our attempt at this challenge taught us a lot about machine learning on real world datasets. We learned that real-world datasets are often tricky to model, as they rarely match the textbook examples of popular machine learning models.

In addition, we gained practical experience in exploring, cleaning, preparing, and validating data. Even data in a structured JSON format must be parsed into a usable format in an efficient manner (until we optimized it, building the dataframe did take a significant amount of time). Checking the accuracy of the model was also an entire process that was equal (if not harder) in effort than preparing the model itself.

Conclusions¶

The Spotify Million Playlist Dataset is an extremely high quality dataset and we thank the researchers at Spotify who curated it and shared it with the public. It was a fantastic experience working with it and although we will not be submitting this to the challenge at the time, this is something we will surely consider in the future as a personal interest and not just as a school assignment.
